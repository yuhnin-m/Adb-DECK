# Core UI Component Guidelines

Документ фиксирует единые правила создания универсальных UI-компонентов в `core/ui`.

## 1. Цель

- Все переиспользуемые UI-элементы создаем в `core/ui`, а не внутри `feature/*`.
- Компонент должен быть пригоден для использования минимум в 2+ фичах.
- API компонента должно быть понятным, типобезопасным и стабильным.

## 2. Структура пакетов

- Каждый новый универсальный компонент создается в **отдельном подпакете**.
- Базовый путь:
  - `core/ui/src/main/kotlin/com/adbdeck/core/ui/<component_name>/`
- Примеры:
  - `.../core/ui/banner/`
  - `.../core/ui/segmented/`
  - `.../core/ui/textfield/`

## 3. Структура файлов компонента

Минимальная рекомендуемая раскладка:

1. `<ComponentName>.kt`  
   Публичные composable-функции и основной UI + все превью компонента (light/dark, основные сценарии).
2. `<ComponentName>Models.kt`  
   Дополнительные модели/enum/option-классы компонента.
3. `<ComponentName>State.kt`  
   Состояние компонента (если есть state-модель).

 

Правило: если появились отдельные модели или state, они **не остаются** в основном файле, а выносятся в отдельные файлы.

## 4. KDoc и комментарии

- Для всех `public` API обязателен KDoc на русском:
  - описание назначения;
  - параметры;
  - поведение по умолчанию;
  - ограничения/нюансы.
- Для `internal/private` сложной логики добавляем краткие комментарии на русском.
- Не писать очевидные комментарии вида “присваиваем переменной значение”.

## 5. Previews

- Для каждого универсального компонента обязательны превью:
  - `Light` тема;
  - `Dark` тема.
- В превью показываем основные состояния компонента:
  - default;
  - selected/active;
  - disabled;
  - error/success/warning/info (если применимо).
- Превью держим в отдельном файле `*Previews.kt`.

## 6. API-дизайн компонента

- Названия должны быть единообразными (`Adb...` префикс для публичных core UI-компонентов).
- Предпочтение параметрам с дефолтами и явной типизацией.
- Взаимоисключающие состояния описываем enum/sealed типами, а не набором Boolean-флагов.
- Никакой feature-специфичной бизнес-логики внутри `core/ui`.

## 7. Стили и токены

- Размеры/отступы/типографика/цвета берутся из `core/designsystem`.
- Избегаем захардкоженных значений, если для них уже есть токен.
- Если токена нет и он нужен нескольким компонентам, добавляем в `core/designsystem`.

### 7.1 Размеры и скругления (только где это полезно)

- Параметры `size` и `cornerRadius` добавляем только для компонентов, где это реально влияет на UX и переиспользование:
  - кнопки, `split-button`, `segmented-button`, чипы, табы, баннеры/карточки.
- Для компонентов-оберток или узкоспециализированных элементов, где размер/скругление не дают пользы, эти параметры в API не добавляем.
- Размеры должны задаваться типизированно (enum/модель размера), а не произвольным `Dp` в публичном API.
- Скругления должны задаваться из общего набора `AdbCornerRadius` (или эквивалентного токена designsystem), без локальных `RoundedCornerShape(8.dp)` в каждом компоненте.
- Значения по умолчанию для радиусов/размеров должны быть консистентны с темой `AdbDeckTheme`.
- Если компонент поддерживает `size`/`cornerRadius`, в превью показываем минимум 2 варианта (например `SMALL` и `LARGE`).

## 8. Интеграция в фичи

- После добавления универсального компонента:
  1. Пилотная миграция в одной фиче.
  2. Проверка визуального результата.
  3. Миграция остальных дубликатов.
- Локальные дубль-компоненты из `feature/*` удаляются после миграции.

## 9. Чеклист перед merge

1. Компонент в отдельном подпакете.
2. Модели/state вынесены в отдельные файлы.
3. KDoc на русском для публичного API.
4. Есть Light/Dark превью.
5. Используются токены designsystem.
6. `./gradlew :core:ui:compileKotlin` проходит.
